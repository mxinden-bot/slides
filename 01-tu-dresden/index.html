HTTPS has won

Encrypted Client Hello (ECH)

  • By default the TLS handshake sends the Server Name Indication (SNI) in clear text, e.g. wikipedia.org.
  • A major attack vector for censors today.
  • ECH encrypts parts of the TLS Client Hello, including the SNI.
  • Public key fetched via the DNS HTTPS record. Especially relevant behind CDNs.
  • Adoption still low. On TCP, Firefox attempts ECH ~0.3% of the time, and it nearly always succeeds.

QUIC and HTTP/3

  • General-purpose transport on top of UDP
  • Encrypted (meta)data
  • 1-RTT connection setup, 0-RTT on resumption
  • Reliable and unreliable delivery, no head-of-line blocking
  • Easy to evolve, often in user space

How does a browser learn a site speaks HTTP/3?

HTTP/3 has to be discovered. Two mechanisms:

Alt-Svc header  RFC 7838

  • An HTTP response header: Alt-Svc: h3=":443".
  • Learned after a full HTTP/1.1 or HTTP/2 connection.
  • So the very first connection is never HTTP/3.

HTTPS DNS record  RFC 9460

  • An HTTPS record carrying alpn="h3".
  • Learned during DNS resolution, before connecting.
  • Also carries IP hints and the ECH key.

The wasted round trip

Alt-Svc only

HTTP/3 only on the second connection.

HTTPS record

HTTP/3 on the first connection.

Save a round trip

  • Many sites send Alt-Svc: h3 but publish no HTTPS record.
  • Every first visit then wastes a round trip: connect over HTTP/2, read the header, upgrade to HTTP/3 only next time.
  • Publish an HTTPS record with alpn="h3" and the client goes straight to QUIC on connection #1.
  • Bonus: the HTTPS record is the only way to deliver ECH.

Oblivious HTTP

  • Similar to MASQUE, but request-oriented, not connection-oriented.
  • So requests are unlinkable: the gateway cannot associate two requests as coming from the same user.
  • The trade-off is performance overhead: the extra relay hop plus per-request encryption cost latency (Oblivious DoH ~228 ms vs ~146 ms for plain DoH, Cloudflare).
  • Used for Telemetry and Oblivious DoH.

Happy Eyeballs v3

A cardinality explosion of protocol combinations to choose from:

  • IPv4 / IPv6
  • TCP + TLS or QUIC
  • DNS A, AAAA or HTTPS records
  • HTTP alt-svc header
  • WebSocket or WebTransport

A set of heuristics to reduce connection-establishment latency while racing candidates.

Congestion control: ECN and L4S

  • Firefox’s QUIC stack does Cubic today; no BBR planned near-term.
  • ECN marking and echoing shipped: 58% of network paths are ECN-capable (GLAM).
  • Already 1 in 5 congestion responses is triggered by an ECN CE mark, not packet loss (GLAM).
  • Modernizing slow start to exit before loss (HyStart++ / SEARCH), instead of overshooting the path (neqo#2858).
  • Ready for an L4S congestion controller on the server side. (Hello, Comcast users.)

What do we not see happening?

  • Old protocols (IPv4, DNS53, HTTP/1 & HTTP/2) going away.
  • IPv6 is ~30% of requests and flat across the last ~10 releases: still ~2 in 3 go over IPv4.
  • An anonymous colleague: “Hopefully no HTTP/4.”

What can you do for a healthy Internet?

  • If you use a browser: enable DoH (Firefox: Settings → Privacy & Security → DNS over HTTPS).
  • If you operate DNS infra: expose a DoH endpoint.
  • If you operate a website: serve it over HTTP/3, publish an HTTPS record, and add ECH.

The Internet of the (near) future

  1. Resolve the site over DNS over HTTPS; its HTTPS record returns IPv6, h3 and ECH.
  2. Connect with a 0-RTT QUIC handshake over IPv6, with ECH keeping the SNI private: no wasted round trip.
  3. Fetch the page over HTTP/3, on that same first connection.
  4. Stay responsive under load: ECN / L4S keep latency near idle even under load.
