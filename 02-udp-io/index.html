Modern UDP I/O for Firefox in Rust

recvmmsg, GRO and GSO across four platforms, with a no-alloc receive path

HTTP Workshop 2026  ·  Max Inden · Mozilla

Why UDP I/O matters for Firefox

  • A large and growing share of Firefox HTTP traffic is HTTP/3 over QUIC, i.e. UDP.
  • QUIC lives in user space; the kernel hands us raw datagrams.
  • We replaced Firefox’s dated NSPR-based UDP I/O with a modern Rust stack.
  • Goal: close the gap to heavily optimized, in-kernel TCP.

Userspace QUIC on UDP is expensive

An un-optimized userspace transport on top of UDP may do:

  • One syscall per datagram
  • < 1500 byte MTU for Internet traffic
  • ACK up to every second packet
  • de Bruijn et al.: “3.5× the CPU cycles per byte”

Segmentation offload: GSO / GRO

  • Linux GSO/GRO, Windows USO/URO.
  • One syscall carries many packets; the NIC/stack splits or coalesces.
  • Firefox Nightly (Linux GRO): P75 read syscall reads ≥2 packets, P95 ≥10.
  • P75 read 2.4 KiB, P95 12 KiB per syscall.
  • Close to the 1 Gbit/s loopback benchmark ceiling.

Multi-message syscalls

  • Linux sendmmsg / recvmmsg.
  • macOS sendmsg_x / recvmsg_x (Apple, undocumented).
  • One syscall moves a batch of independent datagrams.
  • ~11% improvement on an OS loopback transfer.

One syscall, many datagrams.

quinn-udp: one abstraction, four platforms

  • QUIC UDP I/O in Rust via quinn-udp, shared with the Quinn project.
  • Abstracts GSO/GRO, USO/URO, sendmmsg/recvmmsg, sendmsg_x/recvmsg_x behind one API.
  • Integrated into Mozilla’s neqo QUIC stack.
  • Reads auxiliary IP metadata (ECN) from the same control messages.

No-alloc receive path

  • A single, long-lived 64 KiB receive buffer shared across all QUIC connections.
  • No per-datagram allocation on the hot receive path.
  • Soundness of the shared buffer checked at compile time by Rust’s borrow checker.
  • Significant CPU-time reduction on large transfers.
64 KiBone buffer, all connections, zero receive-path allocation

Ancillary win: ECN

  • Modern syscalls expose control messages: we can read IP metadata like ECN.
  • Firefox Nightly: ~50% of paths are ECN-capable outbound.
  • P75 of QUIC connections see ≥ 0.6% CE marks on the receive path.
  • Enables congestion signalling without loss, and readiness for L4S.

Result: < 1 → 4 Gbit/s

OS idiosyncrasies (war stories)

  • Windows: URO fails on ARM64 with WSL (no segment size); USO rolled back over packet loss and driver crashes.
  • macOS: no segmentation offload; sendmsg_x/recvmsg_x undocumented; a 10.15 non-IP edge case.
  • Linux: GSO/GRO first; one socket per connection; sandbox and runtime GSO checks.
  • Android: x86 32-bit uses socketcall; API ≤ 25 returns EINVAL with ECN (retry without); GSO single-segment errors.

What’s next

  • Rolling out PLPMTUD (RFC 8899), ECN, and QUIC ACK frequency.
  • Send-path optimizations: GSO, USO, sendmsg_x, a long-lived send buffer.
  • From Nightly toward Beta and Release.
