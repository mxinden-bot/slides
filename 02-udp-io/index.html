Modern UDP I/O for Firefox in Rust

Closing the gap to in-kernel TCP

HTTP Workshop 2026  ·  Max Inden · Mozilla

Why Firefox rebuilt its UDP I/O

  • ~20% of Firefox HTTP traffic is HTTP/3 over QUIC, i.e. UDP.
  • Firefox’s QUIC implementation lives in user space; the kernel hands us raw datagrams.
  • We replaced Firefox’s dated NSPR UDP I/O (one sendto/recvfrom per datagram) with a modern Rust stack.
  • Goal: close the gap to heavily optimized, in-kernel TCP.

Tracking: Bugzilla 1901292: [meta] Fast UDP for Firefox. Background: max-inden.de/post/fast-udp-io-in-firefox.

Userspace QUIC on UDP is expensive

user space
⇒1 syscall
kernel · NIC
→wire
Internet
  • One syscall per datagram, < 1500 bytes each.
  • ACK up to every second packet.
  • Early QUIC servers spent ~3.5× the CPU cycles per byte of TLS/TCP (Langley et al., SIGCOMM 2017; later optimized toward ~2×).

Segmentation offload: GSO / GRO

user space
⇒1 syscall
kernel · NIC
→NIC splits
Internet
  • Linux GSO/GRO, Windows USO/URO: one syscall carries a big buffer; the NIC splits (send) or coalesces (receive).
  • Real receive coalescing (GRO): median 1 datagram/read, but the P99 read pulls in 14.
  • Median received segment: 1217 bytes.

Multi-message syscalls

user space
⇒1 syscall
kernel · NIC
→wire
Internet
  • Linux sendmmsg / recvmmsg; macOS sendmsg_x / recvmsg_x (Apple, undocumented).
  • One syscall moves a batch of independent datagrams.
  • ~11% improvement on an OS loopback transfer (FOSDEM 2025).

quinn-udp: one abstraction, every platform

  • QUIC UDP I/O in Rust via quinn-udp, shared with the Quinn project.
  • Offload-optimized where Firefox ships: Linux, Windows, macOS, Android.
  • The same abstraction compiles for iOS, FreeBSD, NetBSD, OpenBSD, DragonFly, Solaris, Redox, with a generic fallback path.
  • Reads auxiliary IP metadata (ECN) from the same control messages.

No-alloc receive path

  • A single, long-lived 64 KiB receive buffer shared across all QUIC connections.
  • No per-datagram allocation on the hot receive path.
  • The iterator hands out borrowed slices of that one buffer; Rust’s borrow checker proves there is no aliasing or use-after-free, at compile time.
  • Significant CPU-time reduction on large transfers.

Result

< 1 Gbit/s → 4 Gbit/s

Roughly 4× on CPU-bound transfers. Afterwards, CPU time is dominated by I/O syscalls and cryptography, not per-datagram overhead.

ECN: observed, and acted on

  • Modern syscalls expose IP metadata via control messages, including ECN.
  • 58% of paths are ECN-capable; but 33% bleach the bits (strip ECN in transit).
  • When we do see CE marks, Firefox reacts: classic-ECN congestion response, no packet loss required.

OS idiosyncrasies (war stories)

  • Windows: URO on ARM64 + WSL returns no segment size, disabled (quinn#2041, bug 1916558); USO raised P95 loss to ~6.5% and crashed a driver, rolled back (bug 1979279).
  • macOS: no offload; sendmsg_x/recvmsg_x undocumented, pref-gated and not shipped (neqo#2638); 10.15 unknown address family (quinn#2387).
  • Linux: GSO can pass the probe but fail at send with EIO, resend individually (quinn#2399, bug 2049334).
  • Android: x86 socketcall crashes under seccomp (quinn#1966); API ≤ 25 returns EINVAL with ECN, retry without (quinn#2079); single-segment GSO error (quinn#2050).

Status: it rode the trains

  • The quinn-udp UDP path is on by default beyond Nightly: most of this has shipped to Release (bug 1910360).
  • ECN shipped: marking, echoing, and CE-based congestion response.
  • Held back where the OS fights us: Windows USO/URO stay off; Apple sendmsg_x stays pref-gated.
  • Still to do: send-path offload hardening, and a long-lived send buffer.

Takeaways

One syscall should carry many packets

  • Offload and batching close most of the gap to kernel TCP.
  • The hard part is not the fast path: it is every OS’s edge cases.
  • Rust’s borrow checker made the shared-buffer receive path safe by construction.
