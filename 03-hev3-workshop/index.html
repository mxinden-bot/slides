Rollout of Happy Eyeballs v3 in Firefox

A modern connection-establishment state machine, in production

HTTP Workshop 2026  ·  Max Inden · Mozilla

A cardinality explosion

Connecting to a modern origin means choosing among:

  • IPv4 or IPv6
  • TCP + TLS or QUIC
  • DNS A, AAAA or HTTPS records
  • Alt-Svc header, HTTPS-record ALPN
  • WebSocket or WebTransport

Firefox’s old logic grew organically. We replaced it with one state machine based on the Happy Eyeballs v3 IETF draft.

draft-ietf-happy-happyeyeballs-v3-04, HAPPY WG.

What Happy Eyeballs v3 does

  1. Fire DNS queries together: HTTPS, AAAA, A.
  2. Group and sort endpoints: by protocol + security (ALPN, ECH), then service priority, then RFC 6724, interleaving address families.
  3. Race connection attempts one at a time, staggered; first success wins, the rest are cancelled.

Default preference: IPv6 > IPv4, QUIC > TCP. HTTPS records let QUIC/h3 be discovered at resolution time, no Alt-Svc round trip.

Two timers do the work

Resolution Delay

  • How long to wait for the preferred family / SVCB before committing.
  • Draft default 50 ms. Firefox: 50 ms.

Connection Attempt Delay

  • Stagger between successive attempts.
  • Draft default 250 ms. Firefox: 50 ms with a ×2 multiplier (50, 150, 350, 750 ms …).

The single 250 ms default is too coarse in practice; we ship a smaller base with exponential back-off.

A reusable library: mozilla/happy-eyeballs

  • A deterministic, sans-I/O state machine: no network, no clock. The caller drives all I/O and passes now explicitly.
  • No dependency on Firefox; embeds in Necko’s C++ event loop.
  • Permissive license (MIT OR Apache-2.0). Input and contributions welcome.
  • Tested as scenario scripts against the state machine, plus cargo-mutants mutation testing.

Deterministic, so easy to formalize

#[test]
fn connection_attempt_delay() {
    let (mut now, mut he) = setup();
    // Each tuple is (input event, expected next action):
    he.expect(vec![
        (None, Some(out_send_dns_https(Id::from(0)))),
        (None, Some(out_send_dns_aaaa(Id::from(1)))),
        (None, Some(out_send_dns_a(Id::from(2)))),
        (Some(in_dns_https_positive_no_alpn(Id::from(0))), Some(out_resolution_delay())),
        (Some(in_dns_aaaa_positive(Id::from(1))),          Some(out_attempt_v6(Id::from(3)))),
        (Some(in_dns_a_positive(Id::from(2))),             Some(out_connection_attempt_delay())),
    ], now);
    now += CONNECTION_ATTEMPT_DELAY; // move time forward, next attempt fires
}

How do connections learn about h3?

  • 22% of connections learned h3 only via Alt-Svc: they paid a round trip they didn’t need to.
  • Only 4.7% via an HTTPS record alone; 8% via both.
  • The Alt-Svc-only slice is exactly the gap an HTTPS record would close.

What HTTPS records carry in the wild

Where the implementation goes further

  • The draft interleaves address families; the library also interleaves protocol variants, so QUIC and the other family are both reached within the first attempts.
  • A configurable connection-attempt-delay multiplier (exponential back-off): Firefox runs 50 ms ×2, not the draft’s 250 ms default.
  • A flag to skip waiting for the preferred family.

Implementation choices, not draft changes: practical input for the working group.

Lessons and open questions

  • The 250 ms attempt delay is too coarse: real deployments want a smaller base with back-off.
  • The Alt-Svc-vs-HTTPS-RR gap is measurable and costs round trips: publish HTTPS records.
  • Still open: proxy awareness (MASQUE connect-udp vs HTTP CONNECT ordering) and WebSocket / WebTransport EXTENDED CONNECT.
